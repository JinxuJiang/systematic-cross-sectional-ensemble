# 03.1 设计原理与逻辑架构
> 模型训练层（Alpha Combiner）- AI模型训练与Walk-forward滚动验证

---
## 一、层级职责
### 1.1 核心任务
| 任务 | 说明 | 输出位置 |
|:---|:---|:---|
| 特征工程 | 加载因子数据，构造X(特征矩阵) | 内存DataFrame |
| 标签计算 | 基于真实交易时点计算y(未来收益) | 内存Series |
| 模型训练 | LightGBM回归/LambdaRank排序学习 | experiments/{exp_id}/models/ |
| 滚动验证 | Walk-forward切分，防止数据泄露 | 多Fold验证 |
| **模型融合** | **多模型动态IC加权融合（多周期组合）** | **experiments/{ensemble_id}/** |
| 预测输出 | 生成predictions.parquet供回测层使用 | experiments/{exp_id}/ |

### 1.2 职责边界
- 本层只负责模型训练，不做因子计算（在02层完成）
- 本层不做回测和绩效分析（在04层完成）
- 本层保证训练流程无数据泄露，但不保证模型一定盈利
- 本层输出预测分数，不做任何交易决策

### 1.3 V1版本修复要点
| 问题 | 旧版本 | V1修复 | 影响 |
|:---|:---|:---|:---|
| 数据泄露 | 训练/验证/测试集连续无间隔 | 双重Gap隔离（21天） | 验证集IC更真实（降低）|
| 标签偏差 | 使用T日收盘价计算 | 使用T+1开盘买入，T+(horizon+1)开盘卖出 | 回测收益更贴近实盘 |

### 2.3 Walk-forward滚动机制
```
传统K折交叉验证（金融数据不适用）:
┌────────────────────────────────────────────────────────────────┐
│  Fold 0: [Train ■■■■■■■■■■][Valid □□□][Test ○○○]              │
│  Fold 1: [Valid □□□][Train ■■■■■■■■■■][Test ○○○]              │
│  Fold 2: [Test ○○○][Train ■■■■■■■■■■][Valid □□□]              │
│                                                                │
│  问题: 训练集用了未来数据！                                    │
└────────────────────────────────────────────────────────────────┘

Walk-forward滚动验证（金融数据专用）:
┌────────────────────────────────────────────────────────────────┐
│                                                                │
│  Fold 0: [Train ■■■■■■■■■■][Gap][Valid □□][Gap][Test ○○]      │

│                                                                │

│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 2.4 双重Gap机制详解
```
为什么需要Gap？

场景: horizon=20，使用T+1开盘买入，T+21(即T+(horizon+1))开盘卖出

无Gap（数据泄露）:
┌─────────────────────────────────────────────────────────┐
│ Train: ... 2024-01-30, 2024-01-31                       │
│ Valid: 2024-02-01, 2024-02-02, ...                      │
│                                                         │
│ 问题: Train最后一天(1/31)的标签需要用到2/21的开盘价     │
│       但Valid从2/1开始，Train"看到"了Valid的未来！       │
└─────────────────────────────────────────────────────────┘

有Gap=21（安全）:
┌─────────────────────────────────────────────────────────┐
│ Train: ... 2024-01-30, 2024-01-31                       │
│ Gap:   2024-02-01 ~ 2024-02-29 (21个交易日)             │
│ Valid: 2024-03-01, 2024-03-02, ...                      │
│                                                         │
│ 安全: Train标签用到2/21，Gap覆盖到2/29，Valid从3/1开始  │
│       Train看不到Valid的任何信息                        │
└─────────────────────────────────────────────────────────┘

Gap大小计算:
  Gap = label_horizon + 1 = 20 + 1 = 21天
  原因: T+1开盘买入，持有horizon天，T+(horizon+1)开盘卖出（以20d为例即T+21）
```

### 2.5 回归模型 vs LambdaRank排序模型
| 对比项 | LightGBMModel (回归) | LightGBMRankModel (LambdaRank) |
|:---|:---|:---|
| objective | regression | lambdarank |
| metric | mse/rmse | ndcg |
| 标签处理 | 直接使用收益率 | 分10档离散化 |
| 优化目标 | 预测收益率数值 | 优化股票排名 |
| 输出含义 | 预测收益大小 | 相关性分数（用于排序）|
| 适用场景 | 收益分布相对均匀 | 收益极端值多、需聚焦Top股票 |
| 过拟合风险 | 中等 | 较低（关注相对顺序）|

## 三、输入输出契约
### 3.1 输入规范（来自02因子层）
| 文件 | 路径 | 格式 | 用途 |
|:---|:---|:---|:---|
| 技术因子 | processed_data/factors/technical/*.parquet | 宽表 | 特征X |
| 财务因子 | processed_data/factors/financial/*.parquet | 宽表 | 特征X |
| 收盘价 | processed_data/market_data/close.parquet | 宽表 | 确定股票池 |
| 开盘价 | processed_data/market_data/open.parquet | 宽表 | 计算标签y |

### 3.2 输出规范（predictions.parquet）
| 字段 | 类型 | 说明 |
|:---|:---|:---|
| date | datetime64[ns] | 预测日期（T日）|
| stock_code | str | 股票代码 |
| pred_score | float64 | 模型预测分数（越高越好）|
| actual_return | float64 | 实际未来收益（标签，用于评估）|
| fold_id | int | 所属Fold编号 |

### 3.3 模型文件规范
| 属性 | 规范 |
|:---|:---|
| 格式 | pickle (.pkl) |
| 命名 | model_fold_{fold_id:03d}.pkl |
| 内容 | 完整模型对象（含feature_names） |
| 加载 | BaseModel.load(path) |

### 3.4 特征重要性规范
| 字段 | 类型 | 说明 |
|:---|:---|:---|
| feature | str | 特征名 |
| importance | float64 | Gain重要性 |
| split | int | 分裂次数 |

## 五、核心设计决策
### 5.1 为什么用Walk-forward而非K折交叉验证？
| 方案 | 优点 | 缺点 | 本项目选择 |
|:---|:---|:---|:---|
| K折交叉验证 | 数据利用率高 | 训练集用未来数据，金融数据不适用 | ❌ |
| Walk-forward | 符合时序逻辑，无未来函数 | 早期Fold数据少 | ✅ |
| 组合交叉验证 | 稳健性高 | 复杂度高，实现困难 | ❌ |

理由：
- 金融数据具有时序依赖性，未来信息不能泄露到训练集
- Walk-forward模拟实盘：用历史数据训练，预测未来
- 尽管早期Fold训练数据少，但更符合实际交易场景

### 5.2 为什么标签用开盘价而非收盘价？
| 方案 | 计算方式 | 实盘可执行性 | 本项目选择 |
|:---|:---|:---|:---|
| 收盘价标签 | close[t+20]/close[t] - 1 | 差（T日收盘后才能计算因子） | ❌ |
| 开盘价标签 | open[t+21]/open[t+1] - 1 | 好（T+1开盘买入真实可行） | ✅ |

理由：
- T日收盘后才能拿到完整数据计算因子
- 实盘只能在T+1日开盘下单
- 用收盘价计算有"看到未来"偏差，回测收益虚高

### 5.3 为什么用双重Gap？
| 方案 | gap_train_valid | gap_valid_test | 安全性 |
|:---|:---|:---|:---|
| 无Gap | 0 | 0 | 数据泄露 |
| 单Gap | 21 | 0 | 验证集泄露到测试集 |
| 双重Gap | 21 | 21 | ✅ 安全 |

Gap计算公式：
```
gap = label_horizon + 1 = 20 + 1 = 21天

原因:
- T日收盘后计算因子
- T+1日开盘买入
- 持有20天
- T+(horizon+1)日开盘卖出（以20d为例即T+21）
- 因此标签用到T+(horizon+1)的数据
- Gap必须>=21才能确保不泄露
```

### 5.4 为什么支持两种模型（回归+排序）？
| 模型 | 适用市场状态 | 特点 |
|:---|:---|:---|
| 回归 | 收益分布均匀，无明显极端值 | 直接预测收益数值 |
| LambdaRank | 收益两极分化，头部效应明显 | 聚焦排名，忽略绝对值 |

实践中建议：
- 默认使用LambdaRank，对A股极端行情更稳健
- 回归模型用于分析特征与收益的线性关系
- 可尝试ensemble两种模型

### 5.5 为什么用MultiIndex而非列式存储训练数据？
| 方案 | 优点 | 缺点 | 使用场景 |
|:---|:---|:---|:---|
| MultiIndex | 方便截面操作，保证(date,stock)唯一性 | 内存占用稍高 | 训练过程中 |
| 列式 | 存储高效，下游易读 | 截面操作麻烦 | 最终输出 |

训练时使用MultiIndex：
```python
# 按日期分组计算排名
df.groupby(level=0).rank()

# LambdaRank需要的group信息
dates = X.index.get_level_values(0)
groups = pd.Series(dates).value_counts().sort_index()
```

输出时转列式：
```python
result_df = pd.DataFrame({
    'date': X_test.index.get_level_values(0),
    'stock_code': X_test.index.get_level_values(1),
    'pred_score': predictions,
    ...
})
```

## 六、关键概念解释
### 6.1 IC（信息系数）
**定义**：预测分数与实际收益的相关性

```python
# Pearson IC (线性相关)
ic = predictions.corr(actual_returns)

# Rank IC (Spearman，更稳健，关注排名)
rank_ic = predictions.corr(actual_returns, method='spearman')
```

**解读**：
| IC范围 | 含义 |
|:---|:---|
| > 0.05 | 有效信号 |
| > 0.10 | 优秀信号 |
| 0.02-0.05 | 弱信号，需谨慎 |
| < 0.02 | 无效 |

**注意**：V1修复后IC会比旧版低（因为标签更真实），但这是好事！

### 6.2 Walk-forward窗口配置
```
train_window='3Y'  → 约756个交易日
valid_window='6M'  → 约126个交易日  
test_window='3M'   → 约63个交易日
step='3M'          → 约63个交易日

Fold数量估算:
  总交易日: ~3500天(2010-2024)
  最小需要: 756 + 21 + 126 + 21 + 63 = 987天
  可用长度: 3500 - 987 = 2513天
  Fold数量: 2513 / 63 ≈ 40个

实际约有40-50个Fold，每个Fold约3个月测试期
```

### 6.3 Early Stopping（早停）
**机制**：
```python
model.fit(X_train, y_train, X_valid, y_valid)
# 监控验证集metric，连续N轮无提升则停止
```

**为什么验证集也需要标签？**
- 早停需要验证集metric来判断是否过拟合
- 因此验证集和训练集一样需要计算标签
- Gap确保验证集标签不泄露到训练集

**测试集用途**：
- 不参与训练决策
- 只用于最终评估模型泛化能力
- 模拟完全未知的数据

### 6.4 特征重要性
**计算方法**：
- Gain: 该特征带来的总增益（越高越重要）
- Split: 该特征被用于分裂的次数

**使用注意**：
- 特征重要性会随时间变化（市场状态改变）
- 不应仅依赖重要性进行特征选择
- 高重要性 ≠ 高IC贡献
- 注意多重共线性导致的 importance分摊
