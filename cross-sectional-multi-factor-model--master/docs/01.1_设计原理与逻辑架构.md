定位：从 QMT 下载原始行情与财务数据，存为标准化 Parquet 格式。


---
一、层级定位

1.1 核心职责
数据层是多因子系统的数据基础设施，解决"数据从哪来、怎么存"的问题。

| 职责 | 说明 | 关键设计 |
|:---|:---|:---|
| 行情数据搬运 | 下载全A股日频OHLCV数据 | 等比前复权 |
| 财务数据搬运 | 下载四张财务报表原始数据 | PIT格式存储，记录report_date和m_anntime |
| 元数据维护 | 股票基础信息、行业分类 | 申万一级31个行业映射 |

1.2 设计原则

| 原则 | 具体体现 |
|:---|:---|
| 存算分离 | 数据层只负责搬运和存储，不做任何计算 |
| 增量友好 | 支持月度增量更新，避免重复下载历史数据 |
| 扩张友好 | 方便后续加入新数据源、新接口 |

2.2 文件职责

| 文件 | 类型 | 职责 |

| Base_DataEngine.py | 核心类 | 封装QMT API交互，实现数据下载逻辑 |
| monthly_update.py | 入口脚本 | 增量更新入口，继承DataEngine实现智能合并 |
| data_main.py | 入口脚本 | 全量下载命令行入口，支持--full模式 |

2.3 关键方法

**download_market_data**
- 批量下载行情数据，分批处理（每批300只）
- 使用`dividend_type='front_ratio'`等比前复权
- 失败自动重试3次，超时记录日志继续下一只

**download_financial_data**
- 下载四张表（Income/Balance/CashFlow/PershareIndex）
- 按report_date + m_anntime外连接合并
- 财务报表的PIT处理：修正只保留第一个数据，解决财务报表修正而导致同一个m_anntime存在多个report_date的问题

**monthly_update**（子类方法）
- 行情：全量覆盖（修正复权漂移）
- 财务：智能合并（保留历史 + 追加新数据）
- 元数据：重新下载

三、数据流向

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│  QMT API    │────▶│   DataEngine     │────▶│   raw_data/         │
│             │     │   下载 + 标准化   │     │   本地Parquet存储    │
└─────────────┘     └──────────────────┘     └─────────────────────┘

QMT API调用:
├── get_market_data_ex() ──────→ market_data/{code}.parquet (11列)
│   dividend_type='front_ratio'
│
├── download_financial_data2() → financial_data/{code}.parquet (323列)
│   report_type='announce_time'
│
└── get_stock_list_in_sector() ─→ industry_map.csv / stock_info.parquet
```

存储特点：
- raw_data 行情/财务均按个股分文件存储（长格式），见下文7的架构
- 因子层读取后需转换为宽表（日期×股票）

四、输入输出契约

| 接口 | 用途 | 关键参数 |
|:---|:---|:---|

4.2 下游输出（给因子层）

| 输出 | 路径 | 格式 | 说明 |
|:---|:---|:---|:---|
| 行情数据 | data/raw_data/market_data/{code}.parquet | 长表 | 11字段，等比前复权。按照个股分，5000+文件 |
| 财务数据 | data/raw_data/financial_data/{code}.parquet | 长表 | 323字段，财务报表PIT后。按照个股分，5000+文件 |

| 股票信息 | data/raw_data/stock_info.parquet | Parquet | 基础元数据。一个文件 |

五、关键设计决策

5.1 为什么选Parquet？

| 格式 | 优点 | 缺点 | 结论 |
|:---|:---|:---|:---|
| CSV | 人类可读 | 体积大、无类型、读取慢 | ❌ 不适合海量数据 |
| HDF5 | 读取快 | 兼容性差，版本问题 | ❌ 生态局限 |
| Parquet | 压缩率高、列式读取、类型安全 | 不可直接查看 | ✅ 最佳选择 |

5.2 为什么财务按个股存储？

| 方案 | 读取 | 更新 | 内存 | 结论 |
|:---|:---|:---|:---|:---|
| 大宽表 | 快（单文件） | 慢（重写整个文件） | 占用大 | ❌ 更新成本高 |
| 个股文件 | 需合并 | 快（单文件操作） | 按需加载 | ✅ 更适合低频更新 |

5.3 为什么等比前复权？

| 复权方式 | 计算方式 | 长期表现 | 收益率计算 |
|:---|:---|:---|:---|
| 普通前复权 | 加减法 | 可能出现负数 | 需特殊处理 |
| 等比前复权 | 乘法 | 永远为正 | close.pct_change()即可 |

- 前复权是以现在的价格为基准，把历史价格根据分红、除权调整。适合回测
- 后复权适合看盘

5.4 数据更新时为什么行情全量+财务合并？

| 数据类型 | 特性 | 更新策略 |
|:---|:---|:---|
| 行情数据 | 复权价格随时间漂移（新分红会改变历史价格） | 必须全量覆盖 |
| 财务数据 | 发布后静态不变 | 智能合并即可 |

六、数据格式规范

6.1 行情数据（11列）

| 字段 | 类型 | 说明 | 下游用途 |
|:---|:---|:---|:---|
| time | int64 | 毫秒时间戳 | 需转换为datetime |
| open/high/low/close | float64 | 等比前复权价格 | 核心价格数据 |
| volume | int64 | 成交量（股） | 流动性因子 |
| amount | float64 | 成交额（元） | 流动性因子 |
| preClose | float64 | 昨收价格 | 计算ret1 |
| suspendFlag | int64 | 0=正常, 1=停牌 | 过滤停牌日 |
| settelementPrice | float64 | 结算价（期货字段） | 可忽略 |
| openInterest | float64 | 持仓量（期货字段） | 可忽略 |

6.2 财务数据（关键字段）

| 字段 | 说明 | PIT用途 |
|:---|:---|:---|
| report_date | 报告期（如20200331） | 同比环比计算 |
| m_anntime | 公告日 | PIT对齐关键字段 |
| net_profit_excl_merged_int_inc | 归母净利润（累计） | TTM计算基础 |
| revenue_inc | 营业收入（累计） | TTM计算基础 |
