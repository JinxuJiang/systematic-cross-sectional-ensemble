#### 1.2 MonthlyDataUpdater 类

继承自 DataEngine，实现智能增量更新。

**monthly_update 参数**

| 参数 | 类型 | 默认值 | 说明 |
|:---|:---|:---|:---|
| market_start | str | '20100101' | 行情起始日期 |
| financial_start | str | '20230101' | 财务起始日期（近3年） |

**更新策略对比**

| 数据类型 | 策略 | 原因 | 耗时 |
|:---|:---|:---|:---|
| 行情 | 全量覆盖 | 复权价格漂移需修正 | 2-4小时 |
| 财务 | 智能合并 | 历史数据静态不变 | 10-20分钟 |
| 元数据 | 重新下载 | 股票列表会变化 | <1分钟 |

**完整更新流程**

1. 获取最新股票列表（全A股）
2. 下载行情数据（全量覆盖）
3. 下载并合并财务数据（智能合并）
4. 下载元数据
5. 记录更新日志到 update_log.json

#### 2.2 download_financial_data

**参数说明**

| 参数 | 类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| stock_list | list | 是 | 股票代码列表 |
| start_time | str | 是 | 开始日期，YYYYMMDD |
| end_time | str | 否 | 结束日期，默认最新 |

**实现流程**

```
分批触发下载 → 等待缓存写入 → 逐只获取数据
                                           ↓
                 四表外连接合并 → 数据清洗 → 保存parquet
```

**四表合并逻辑**

| 表名 | 内容 | 合并方式 |
|:---|:---|:---|
| Income | 利润表 | outer join |
| Balance | 资产负债表 | outer join |
| CashFlow | 现金流量表 | outer join |
| PershareIndex | 每股指标表 | outer join |

**关键清洗步骤（两步清洗法）**

基于数据验证：QMT财务数据无快照/预告（不存在 `ann < report` 的情况），简化清洗逻辑如下：

```
步骤0: groupby([report_date, m_anntime]).first()  # 合并四表外连接后的重复行
    
清洗1: 同一 report_date 保留最早 m_anntime      # 防未来函数：丢弃修正公告，只保留第一次发布的正式财报
    
清洗2: 同一 m_anntime 保留最大 report_date      # 修正处理完了之后，处理同天多正式报告（年报+次年一季报）
```

| 步骤 | 操作 | 目的 | 关键决策 |
|:---|:---|:---|:---|
| 0 | 合并同一(report_date, m_anntime)的多行 | 四表外连接后去重 | groupby.first() 取首个非空值 |
| 1 | 按 report_date 排序，保留最早 m_anntime | 防未来函数 | 同一报告期只用首次公告的数据，丢弃后续修正 |
| 2 | 按 m_anntime 排序，保留最大 report_date | 处理同天多报告 | 年报和一季报同一天发布时，保留一季报（报告期更新）|

- 原始数据中不包含快照、预告数据。只包括正式的财务报告、修正。

**实现位置**

- Base_DataEngine.download_financial_data()：下载时即做两步清洗
- MonthlyDataUpdater.download_financial_data_with_merge()：合并新旧数据后重新执行完整两步清洗
- PITAligner.align()：数据层已清洗，因子层只需对齐，无需再清洗

## 三、开发注意事项（重难点）

### 3.1 公告日 vs 报告期的选择

**概念区分**

| 字段 | 含义 | 示例 |
|:---|:---|:---|
| report_date | 财报统计截止日期 | 2024一季报 = 20240331 |
| m_anntime | 财报实际公告日期 | 可能在 20240425 |

**选择原则**

| 场景 | 使用字段 | 原因 |
|:---|:---|:---|
| PIT对齐 | m_anntime | 公告前看不到数据 |
| 同比、TTM计算 | report_date | 确保同一报告期 |

**后果警示**

用report_date做PIT对齐会引入未来函数在2024年3月31日就能"看到"4月25日才公告的财报。

### 3.2 重复值处理（两步清洗法）

**问题背景**

经数据验证，QMT财务数据不存在快照/预告（无 `ann < report` 的情况）。原始数据存在两种重复：
1. 同一报告期的多次公告（同一 `report_date`，不同 `m_anntime`）：
  - 修正：正式报告 → 修正公告
  - 预告：预告 → 正式报告 → 修正公告（预告这个不需要考虑）
2. 同一公告日的多份财报（同一 `m_anntime`，不同 `report_date`）：
  - 同时发布两个正式报告：年报和次年一季报同一天发布
  - 修正：正式报告 → 修正公告

| 步骤 | 目的 | 操作逻辑 |
|:---|:---|:---|
| 清洗1 | 防未来函数：同一报告期保留首次公告 | 按 (report_date, m_anntime) 排序，对相同 report_date 保留第一条（最早ann） |
| 清洗2 | 处理同天多报告：保留最新报告期 | 按 (m_anntime, report_date) 排序，对相同 m_anntime 保留 report_date 最大的 |

**为什么先清洗1再清洗2？**

必须先确保每个报告期只有一个版本（取最早公告，防未来函数），再处理同一天的多份财报（取最新报告期）。顺序颠倒会导致同一报告期的修正版覆盖初版，引入未来函数。

**为什么取首次公告（而非最新修正）？**

防止未来函数、方便：
- 4月24日公告正式报告52亿，5月10日修正为51亿。在4月24日当天，只能看到52亿（正式报告）
- 如果在4月24日使用5月10日修正的51亿，就用了未来才知道的信息

**代码实现**

```python
# 步骤0: 合并四表外连接后的重复行
df = df.groupby(['report_date', 'm_anntime']).first().reset_index()

# 清洗1: 同一 report_date 保留最早 m_anntime（防未来函数）
df = df.sort_values(['report_date', 'm_anntime'])
df = df.drop_duplicates(subset=['report_date'], keep='first')

# 清洗2: 同一 m_anntime 保留最大 report_date（处理同天多报告）
df = df.sort_values(['m_anntime', 'report_date'])
df = df.drop_duplicates(subset=['m_anntime'], keep='last')
```

**实际案例（无快照数据）**

```
# 某股票原始财报数据（QMT无快照，所有 ann >= report）

 report_date   m_anntime   净利润    说明       

  20240331     20240424     52亿    Q1正式报告   清洗1保留（同report_date最早ann）
  20240331     20240510     51亿    Q1修正报告   清洗1丢弃
  20241231     20250424    200亿    2024年报     清洗2丢弃（同m_anntime，保留report_date大的）
  20250331     20250424     55亿    2025Q1       清洗2保留


# 清洗后保留：
# - 20240331, 20240424, 52亿（Q1正式报告）
# - 20250331, 20250424, 55亿（2025Q1，同天发布年报+一季报，保留报告期更新的）
#
# 原因：
# - 清洗1：对于report_date=20240331，保留最早ann=20240424（正式报告），丢弃5/10修正
# - 清洗2：对于m_anntime=20250424，保留最大report_date=20250331（2025Q1），丢弃年报
```

### 3.3 时间格式转换

**行情数据时间字段**

| 属性 | 说明 |
|:---|:---|
| 存储格式 | int64 毫秒时间戳（UTC时间） |
| 示例 | 1704067200000 = 2024-01-01 00:00:00 UTC |
| 转换后 | 北京时间（UTC+8） |

**转换方法**

```python
import pandas as pd

# 方法1：pandas批量转换（推荐）
df['datetime'] = pd.to_datetime(df['time'], unit='ms', utc=True).dt.tz_convert('Asia/Shanghai')

# 方法2：先转UTC再转北京（显式时区处理）
df['datetime'] = pd.to_datetime(df['time'], unit='ms', utc=True).dt.tz_convert('Asia/Shanghai')

# 方法3：去除时区（如果后续不需要时区-aware操作）
df['datetime'] = pd.to_datetime(df['time'], unit='ms', utc=True).dt.tz_convert('Asia/Shanghai').dt.tz_localize(None)
```

**重要说明**

- QMT返回的 `time` 字段是 UTC 毫秒时间戳
- 使用 `pd.to_datetime(unit='ms')` 默认得到的是 UTC 时间
- 需要显式转换为北京时间（Asia/Shanghai）
- 如果不转换，显示的时间会比北京时间慢8小时

### 3.4 前复权漂移问题

**问题描述**

等比前复权价格 = 原始价格 × 复权系数。每次分红除权后复权系数改变，导致历史价格变化。

**影响**

今天下载的2020年收盘价 ≠ 上个月下载的2020年收盘价（差异几分钱）。

**解决方案**

每月全量重新下载行情数据，覆盖所有历史记录。

### 3.5 停牌数据处理

**识别特征**

- volume = 0
- suspendFlag = 1
- close保留停牌前最后价格（非NaN）

**处理建议**

不要删除停牌记录，计算时过滤：
```python
mask = (df['suspendFlag'] == 0) & (df['volume'] > 0)
df_active = df[mask]
```

### 3.6 财务数据缺失值

**现象**

打开财务文件大量NaN，正常。

**原因**

- 很多字段就是没有值。
- 很多股票在整体数据初期没有上市.

**处理原则**

不要试图填满所有NaN。下游按需读取对应字段即可。
