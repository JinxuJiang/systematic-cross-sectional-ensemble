# 00.2 项目文档架构
> 本文档定义整个多因子项目的文档体系，包括各层级文档的命名规范、内容结构和存放位置。
---
## 一、文档体系总览
本项目采用双轨制文档体系：
| 位置 | 用途 | 读者 | 内容特点 |
|:---|:---|:---|:---|
| `docs/` | 放飞书 | 给自己看 | 详细设计、思考过程、踩坑记录，可随意修改 |
| `XX数据/README.md` | 放 GitHub | 给代码看 | 快速上手、目录结构、核心接口 |
---
## 二、文档命名规范
### 2.1 编号规则
| 编号 | 含义 | 示例 |
|:---|:---|:---|
| 00.x | 项目级文档（跨层级） | 00.1_项目总体要求.md |
| 01.x | 数据层文档 | 01.1_设计原理与逻辑架构.md |
| 02.x | 因子层文档 | 02.1_设计原理与逻辑架构.md |
| 03.x | 模型层文档 | 03.1_设计原理与逻辑架构.md |
| 04.x | 回测层文档 | 04.1_设计原理与逻辑架构.md |
### 2.2 后缀规则
每个层级固定三个文档：
| 后缀 | 文档名称 | 内容定位 |
|:---|:---|:---|
| .1 | 设计原理与逻辑架构 | 目的、架构、数据流、设计决策 |
| .2 | 工程实现与规范 | API、数据规范、开发注意事项 |
| .3 | 运维与变更日志 | 检查点、性能、TODO、变更记录 |

## 三、各层级文档详细定义
### 3.1 数据层（01数据/）
位置：
- 飞书文档：`docs/01.1_设计原理与逻辑架构.md`、`docs/01.2_工程实现与规范.md`、`docs/01.3_运维与变更日志.md`
- GitHub 文档：`01数据/README.md`

01.1 设计原理与逻辑架构内容模块：
1. 层级职责：数据层的核心任务、负责的数据类型（行情、财务、元数据）
2. 代码架构：Base_DataEngine 类设计、关键方法说明
3. 数据流向：从 QMT API 到本地 Parquet 的完整流程
4. 输入输出契约：上游 QMT API 接口、下游给因子层的文件格式
5. 代码模块树：文件夹结构、文件职责
6. 核心设计决策：为什么用 Parquet、为什么财务按个股存、为什么等比前复权
7. 关键字段说明：行情 11 个字段、财务关键字段的含义和用途
8. 潜在问题与改进点：路径硬编码、时间戳格式、期货字段残留

01.2 工程实现与规范内容模块：
1. 核心 API 详细说明：每个方法的参数、实现细节、使用示例
   - download_market_data：分批策略、重试机制、复权处理
   - download_financial_data：四表合并、清洗逻辑、去重策略
   - monthly_update：智能更新策略、行情全量覆盖原因
2. 数据规范：行情/财务/行业映射的字段定义、类型、可空性
3. 开发注意事项（难点）：
   - 公告日 vs 报告期的选择（PIT 对齐）
   - 重复值处理（第一次发布原则）
   - 前复权漂移问题
   - 时间格式转换
   - 停牌数据处理
   - 财务数据缺失值
4. 配置说明：QMT 服务器、数据路径、下载参数
5. 快速命令：常用命令行操作
6. 常见问题排查：Q&A 形式的问题诊断

01.3 运维与变更日志内容模块：
1. 关键检查点：数据完整性检查项、检查方法、通过标准
2. 性能基准：全量下载耗时、增量更新耗时、存储空间占用
3. 运维操作步骤：首次初始化、每日增量、月度全量、故障回滚
4. 变更日志：日期-类型-简述-详情-影响-兼容性
5. TODO List：待优化事项（高/中/低优先级）

## 五、文档格式规范
### 5.1 紧凑原则
- 标题后直接跟内容，不加空行
- 段落之间最多一个空行
- 表格前不加空行
- `---` 分隔符前后不加空行

### 5.2 表格格式
- 使用 `|:---|:---|:---|` 分隔行
- 第一列不加粗（不要用 `**字段**`）
- 单元格内容简洁

正确示例：
```markdown
| 字段 | 类型 | 说明 |
|:---|:---|:---|
| time | int64 | 毫秒时间戳 |
| close | float64 | 收盘价 |
```

错误示例：
```markdown
| **字段** | **类型** | **说明** |
|:---|:---|:---|
| **time** | int64 | 毫秒时间戳 |
```

### 5.3 代码块
- 标明语言类型
- 关键行添加注释
